Introdução ao Docker

Ambiente Virtualizado

Diga adeus ao "Na minha máquina funciona" 🚀

O Docker é uma plataforma de virtualização em nível de sistema operacional (containers) que permite empacotar uma aplicação e todas as suas dependências em um único pacote isolado chamado container.

Minha App + Dependências

Isolamento Completo

Cada container roda de forma isolada do sistema hospedeiro e de outros containers. Eles compartilham o Kernel do sistema, mas têm arquivos, variáveis e processos separados.

Leve e Rápido

Como não rodam um sistema operacional completo dentro de si, iniciam em milissegundos e utilizam pouquíssima memória comparados com Máquinas Virtuais.

Consistência

O mesmo container rodando no seu computador pessoal rodará de forma exatamente igual no servidor de produção ou nuvem.

Para entender o Docker, precisamos entender a grande diferença entre a virtualização tradicional (Máquinas Virtuais) e a virtualização baseada em containers.

Máquina Virtual (VM)

Como o Docker consegue rodar aplicações de forma tão isolada compartilhando o mesmo Kernel? O segredo está em recursos nativos do próprio Kernel Linux:

Namespaces

Cria uma parede visual. Ele delimita o que um container consegue enxergar do sistema. Ele faz o container acreditar que tem seu próprio sistema exclusivo.

Máquina Host
PID 1: Systemd
PID 45: Chrome
PID 102: Node.js (App)
PID 105: Nginx (App)
Visão do Container
PID 1: Node.js (App)
PID 2: Nginx (App)
PID 45: Chrome
O container enxerga a si mesmo como o PID 1 (o início de tudo), embora no host ele seja apenas mais um processo comum. Tipos de Namespaces: PID (Processos), NET (Redes), MNT (Pontos de Montagem), IPC (Comunicação), UTS (Hostnames), USER (Usuários).

Control Groups (cgroups)

Cria uma parede de recursos. Ele define o quanto o container pode consumir da máquina host. Evita que um container consuma toda a CPU ou memória RAM e derrube a máquina.

Container: 10%
Outros Processos: 40%
Com os cgroups, garantimos que mesmo sob ataque ou falha de código (como loop infinito), o container nunca extrapolará seus limites alocados.

O fluxo de trabalho do Docker gira em torno de 4 conceitos fundamentais que transformam código fonte em uma aplicação rodando em qualquer lugar do mundo.

1

Dockerfile

É a receita de bolo. Um arquivo de texto com instruções sequenciais que descrevem como construir o ambiente de execução da aplicação.

FROM node:alpine
WORKDIR /app
COPY . .
RUN npm install
CMD ["node", "app.js"]
2

Imagem

É o bolo congelado. Um arquivo estático, somente leitura, gerado a partir do Dockerfile. Contém o código, bibliotecas e dependências prontos.

Camada: Código da App Camada: Dependências npm Camada: Node runtime Camada: SO Alpine Linux
3

Registro

É o supermercado de bolos. Uma nuvem pública ou privada onde você guarda e compartilha suas imagens criadas. Ex: Docker Hub, AWS ECR.

docker pull minha-imagem:latest
4

Container

É o bolo servido e pronto para comer. A imagem em execução. Um container é uma instância ativa de uma imagem.

Container: Online - Porta 8080

Guia Completo de Comandos Docker 📋

Domine o ciclo de vida completo de containers, desde o build até a publicação e descarte de recursos locais ou remotos.

1. Criação & Tagging de Imagens

docker build -t usuario/app:1.0 . Build

O que faz: Constrói uma imagem baseada nas instruções do Dockerfile localizado no diretório atual (.) e atribui a tag usuario/app:1.0.

docker tag local-app:latest usuario/app:1.0 Tag

O que faz: Cria uma referência ou alias com o formato do Docker Hub (usuario/repositorio:tag) a partir de uma imagem local existente.

2. Publicação & Distribuição (Docker Hub)

docker login Auth

O que faz: Realiza a autenticação de segurança no registro (por padrão, o Docker Hub) usando o seu usuário e senha de acesso.

docker push usuario/app:1.0 Push

O que faz: Envia a imagem local para o seu repositório no Docker Hub, deixando-a disponível para download público ou privado.

docker pull usuario/app:1.0 Pull

O que faz: Baixa do repositório remoto a imagem especificada para a sua máquina local (sem executá-la).

3. Execução & Mapeamento

docker run -d -p 8080:80 --name webapp app Background

O que faz: Cria e inicia o container em segundo plano (-d), mapeia a porta do host para a interna (-p 8080:80) e define o nome amigável webapp.

docker run -it --rm -e ENV=prod ubuntu bash Interactive

O que faz: Inicia de forma interativa (-it), injeta uma variável de ambiente (-e), executa o shell bash e destrói o container automaticamente ao sair (--rm).

4. Interação & Inspeção

docker exec -it webapp bash Exec

O que faz: Abre um terminal interativo (-it) rodando o shell bash dentro de um container que já está ativo na máquina.

docker logs webapp Logs

O que faz: Exibe a saída padrão (stdout/stderr) do container. Mostra tudo que a aplicação imprimiu no console.

docker inspect webapp Inspect

O que faz: Retorna uma estrutura JSON detalhada contendo todas as configurações, volumes e redes internas do container.

5. Monitoramento & Estado

docker ps Ativos

O que faz: Lista todos os containers que estão ativos e rodando neste momento na máquina.

docker ps -a Histórico

O que faz: Mostra o histórico de todos os containers criados (ativos, parados ou que sofreram erro de execução).

docker stats Stats

O que faz: Exibe em tempo real o consumo de recursos (CPU, Memória RAM, I/O de disco e rede) de todos os containers ativos.

6. Descarte & Limpeza

docker stop webapp Stop

O que faz: Envia um sinal de encerramento amigável (SIGTERM) para parar de rodar a aplicação interna do container.

docker rm webapp Remover Container

O que faz: Exclui do sistema o container. Ele precisa estar parado para que este comando funcione.

docker rmi app:1.0 Remover Imagem

O que faz: Exclui a imagem do seu HD local. Só funciona se nenhuma instância de container estiver baseada nela.

docker system prune -a Prune Completo

O que faz: Executa uma limpeza total no disco, apagando todos os containers parados, redes não usadas e imagens em cache locais.

7. Orquestração com Compose

docker compose up -d Up

O que faz: Lê o arquivo docker-compose.yml do diretório, baixa/compila as imagens necessárias e inicia todos os serviços em background.

docker compose down Down

O que faz: Para e remove de forma coordenada todos os containers, redes virtuais e volumes associados criados pelo docker compose up.

O Docker ganha superpoderes através de suas flags de comando. Aqui explicamos como funciona o mapeamento de portas, a interatividade no terminal, os processos internos e apresentamos um guia completo das flags essenciais.

Como funcionam as portas (-p)

Containers rodam em redes virtuais isoladas. Para acessá-los de fora (e.g., no seu navegador do host), você precisa mapear uma porta da máquina host para a porta do container.

Host OS (Seu PC)
Porta 8080
Container (Docker)
Porta 80
docker run -p 8080:80 nginx
Isto cria um túnel: requisições na porta 8080 do seu PC são enviadas para a porta 80 do Nginx dentro do container.

Como interagir com um container (-it)

Para interagir com um container e rodar comandos diretamente no terminal dele, usamos as flags:

  • -i (interactive): Mantém o canal de entrada (STDIN) aberto para digitação.
  • -t (tty): Aloca um terminal virtual (pseudo-TTY) para simular o terminal Linux.
Entendendo o comando: docker run -it ubuntu bash

Cria um container a partir da imagem oficial do Ubuntu, conecta seu teclado/terminal a ele e roda o shell bash. Você entra "dentro" do container Ubuntu.

Criação de imagem e Processos

Criando uma imagem: Usamos docker build -t minha-imagem .. A flag -t serve para dar uma tag/nome à imagem e o ponto . indica o contexto/pasta onde o Dockerfile e arquivos estão localizados.

Processos no container: Um container roda em cima do OS host, mas possui processos isolados. O container só fica ativo enquanto seu processo principal (PID 1) estiver ativo. Se você rodar docker run -it ubuntu bash e digitar exit, o processo bash (PID 1) termina e o container desliga.

Dicionário das Flags mais Utilizadas

As flags modificam o comportamento do comando docker run. Abaixo estão as mais comuns no dia a dia:

Flag Nome Completo O que faz Quando usar Exemplo de Uso
-d --detach Executa o container em segundo plano (background) e libera o terminal. Servidores Web, bancos de dados e serviços persistentes. docker run -d nginx
-p --publish Conecta uma porta da máquina host a uma porta interna do container. Quando precisa acessar o serviço pelo navegador (ex: localhost:8080). docker run -p 8080:80 nginx
-it -i --tty Mantém o terminal interativo conectado para digitação. Para explorar pastas ou debugar executando comandos manuais. docker run -it ubuntu bash
--name --name Atribui um nome customizado e único ao container. Para evitar gerenciar o container através de hashes hexadecimais longos. docker run --name web-nginx nginx
--rm --rm Remove o container automaticamente assim que ele finaliza. Para tarefas temporárias, testes rápidos ou scripts de execução única. docker run --rm hello-world
-v --volume Compartilha pastas entre o computador host e o container. Para desenvolvimento local (hot-reload) ou salvar dados persistentes. docker run -v $(pwd):/app node
-e --env Define variáveis de ambiente no container em tempo de execução. Passar chaves de APIs, credenciais ou variáveis de configuração dinâmicas. docker run -e DB_PASS=123 mysql
--restart --restart Define políticas de reinicialização automática do container. Garantir que serviços voltem a rodar se caírem ou após reboot do PC. docker run --restart unless-stopped nginx

Desenvolver com Docker envolve escrever arquivos de configuração para automatizar o ambiente. Abaixo explicamos as linguagens, regras obrigatórias, boas práticas e a sintaxe de cada arquivo essencial.

Instalação e Componentes: O que roda na sua máquina

Ao instalar o Docker Desktop (no Windows/macOS) ou o Docker Engine (no Linux), a sua máquina passa a rodar os seguintes componentes essenciais que trabalham de forma coordenada:

Docker Daemon (dockerd) O serviço em segundo plano (background) que gerencia os containers, baixa as imagens, gerencia redes virtuais e aloca volumes de disco.
Docker CLI (docker) O utilitário de linha de comando que você digita no terminal. Ele se comunica via API REST com o Docker Daemon para executar as suas ações.
Docker Compose CLI Plugin Ferramenta orquestradora que permite rodar aplicações multicontainer complexas descritas em um único arquivo de configuração YAML.
containerd O motor de execução leve de containers de padrão industrial, responsável por gerenciar o ciclo de vida completo de cada container de forma segura.

Linguagem e Estrutura

Dockerfile: Não usa uma linguagem de programação genérica. Possui sintaxe declarativa própria contendo instruções em letras maiúsculas (ex: FROM, COPY) seguidas de parâmetros. É executado sequencialmente de forma compilada.

docker-compose.yml: Escrito em formato YAML. O YAML usa pares de chave: valor e não utiliza delimitadores como chaves {}. Toda a organização hierárquica é feita estritamente através do recuo de espaços.

Regras Obrigatórias e Boas Práticas

  • Regra Obrigatória do YAML: Apenas espaços são permitidos para identação. Pressionar a tecla TAB causará um erro de compilação. Use 2 espaços extras para declarar propriedades filhas.
  • Regra do Dockerfile: A primeira instrução deve obrigatoriamente ser a tag FROM, especificando a imagem que servirá de alicerce.
  • Use o .dockerignore: Equivalente ao .gitignore. Ele impede o envio de arquivos e pastas pesadas ou confidenciais (ex: node_modules/, logs, chaves locais) para dentro do container durante o build.
  • Ordem inteligente das camadas: Instruções pesadas que mudam pouco devem ficar no início para aproveitar a otimização de cache do Docker.

docker build -t minha-web-app .

Verifique se a imagem estática foi criada e está salva localmente:

docker images
3. Execute o Container

Agora instancie a imagem criando um container rodando em segundo plano na porta 8080:

docker run -d -p 8080:80 minha-web-app

Verifique o status da sua aplicação rodando em tempo real:

docker ps
